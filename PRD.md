## 1. 제품 개요

### 1.1 비전 (Vision)

**"가사 노동의 정신적 부담을 줄이고, 가족 간 협력과 공정성을 증진시키는 필수 생활 플랫폼"**

ChoreQuest는 단순한 할 일 관리 앱이 아니라, 가정 내 '정신적 노동(Mental Load)'을 분산시키고 가사 분담의 공정성을 가시화하여 가족 간 갈등을 해소하는 협업 플랫폼입니다.

### 1.2 제품 포지셔닝

| 구분 | 설명 |
|------|------|
| **타겟 시장** | 한국 맞벌이 부부, 유자녀 가정, 룸메이트 |
| **핵심 가치** | 정신적 노동 경감, 공정한 분담, 지속적 동기 부여 |
| **차별화 요소** | 초월적 현지화(카카오톡 연동, 한국형 템플릿), 다층적 게임화 |
| **경쟁 우위** | Sweepy(글로벌) 대비 한국 시장 특화, 가사함께(국내) 대비 정교한 게임화 |

### 1.3 핵심 성공 지표 (KPI)

#### Phase 1 (MVP - 현재)
- 월 활성 사용자(MAU): 1,000명
- 사용자 유지율(Day 7): 40%
- 가구당 평균 집안일 완료율: 60%

#### Phase 2 (Firebase 마이그레이션 후)
- 월 활성 사용자(MAU): 10,000명
- 사용자 유지율(Day 30): 50%
- 유료 전환율: 5%
- 가구당 일일 활성 시간: 5분

## 2. 문제 정의 및 사용자 니즈

### 2.1 해결하고자 하는 핵심 문제

#### 문제 1: 정신적 노동의 불균형
- **현상**: 집안일의 계획, 분배, 감독 업무가 특정인(주로 여성)에게 집중
- **결과**: 인지적/감정적 소진, 가족 간 갈등
- **데이터**: 맞벌이 가정에서도 여성이 정신적 노동의 70% 이상 담당

#### 문제 2: 기여도의 비가시성
- **현상**: "누가 더 많이 했는가"에 대한 주관적 인식 차이
- **결과**: 제로섬 게임 구도, 인정받지 못하는 느낌
- **사례**: "설거지 한 번 하고 '내가 다 하고 있다'고 생각하는 배우자"

#### 문제 3: 지속적 동기 부여의 부족
- **현상**: 반복적이고 인정받지 못하는 노동에 대한 의욕 저하
- **결과**: 집안일 방치, 가정 환경 악화
- **요구**: 성취감과 보상을 느낄 수 있는 시스템

### 2.2 타겟 사용자 페르소나

#### 페르소나 1: 지혜 (32세, 워킹맘)
- **직업**: IT 기업 마케터
- **가족**: 남편(34), 딸(5세)
- **페인 포인트**:
  - 집안일 계획을 혼자 다 세워야 하는 스트레스
  - 남편이 "뭐 도와줄까?"라고 물을 때마다 느끼는 감정 노동
  - 육아+가사+직장의 멀티태스킹으로 인한 번아웃
- **니즈**:
  - 자동으로 스케줄링되고 순환 배정되는 시스템
  - 각자의 기여도를 객관적으로 보여주는 데이터
  - 남편이 주도적으로 참여하도록 유도하는 메커니즘

#### 페르소나 2: 민수 (29세, 신혼 남편)
- **직업**: 스타트업 개발자
- **가족**: 아내(28)
- **페인 포인트**:
  - 집안일을 어떻게 나눠야 공평한지 기준이 모호함
  - 아내에게 "뭐 할까?"라고 물어보는 게 오히려 스트레스를 주는 것 같음
  - 집안일이 지루하고 재미없어서 미루게 됨
- **니즈**:
  - 명확하게 정의된 역할과 담당 업무
  - 집안일을 게임처럼 재미있게 만드는 요소
  - 기여도를 인정받을 수 있는 시각적 피드백

#### 페르소나 3: 수진 & 영미 (26세, 룸메이트)
- **상황**: 대학원생, 원룸 공동 거주
- **페인 포인트**:
  - 청소 차례를 구두로 정하다 보니 혼선 발생
  - "내가 더 많이 하고 있다"는 불만
- **니즈**:
  - 간편한 작업 배정 및 순환 시스템
  - 공정성을 증명할 수 있는 기록

### 2.3 경쟁사 분석 요약

| 경쟁사 | 강점 | 약점 | ChoreQuest 차별점 |
|--------|------|------|------------------|
| **Sweepy** | 효과적인 게임화, 청결도 시각화 | 초기 설정 부담, 한국 시장 비최적화 | 카카오톡 연동, 한국형 템플릿, 협업적 온보딩 |
| **OurHome** | 다양한 기능(쇼핑리스트 등) | 심각한 기술적 불안정성, 나쁜 UX | 안정성 최우선, 사용자 중심 설계 |
| **가사함께** | 한국 시장 특화, 균형 메타포 | 제한된 게임화, 작은 사용자 기반 | 정교한 다층 게임화, 리더보드+협력 목표 |
| **Todoist/TimeTree** | 강력한 생산성 기능 | 가사 관리 특화 부재, 정신적 노동 경감 불가 | 가사 전용 기능, 자동 스케줄링, 담당자 순환 |

#### 3.2.2 정신적 노동 경감 UX (P0)

##### 협업적 온보딩
```
플로우:
1. 가구 생성자가 앱 설치
2. 초대 링크/코드 생성
3. 가족 구성원이 함께 앱 설정
4. 공동으로 집안일 목록 작성
   - "쓰레기 배출은 보통 누가 담당하나요?"
   - "화장실 청소는 얼마나 자주 해야 할까요?"
5. 역할 분담 합의
```

**설계 원칙**:
- 한 사람이 혼자 설정하지 않도록 강제
- 앱을 '중립적인 제3자'로 포지셔닝
- 초기부터 공동 책임감 부여

##### 자동 스케줄링 및 순환
- 주기 기반 자동 생성 (매일, 매주, 격주, 매월)
- 담당자 자동 순환 (Round-robin)
- "오늘의 할 일" 자동 큐레이션
- 집안일 템플릿 시스템

##### 관리자 역할 순환 (프리미엄)
- 새 집안일 추가 권한
- 완료 승인 권한
- 역할이 주간 단위로 성인 구성원 간 자동 순환
- "이번 주 관리자: 지혜님" 표시

## 5. 비기능 요구사항 (NFR)

### 5.1 성능
- **앱 초기 로딩**: 3초 이내
- **집안일 완료 반응 시간**: 500ms 이내
- **실시간 동기화 지연**: 2초 이내
- **월간 활성 사용자 지원**: 10,000명 (Phase 2 목표)

### 5.2 안정성
- **목표 가동시간(Uptime)**: 99.5%
- **크래시 발생률**: < 1%
- **데이터 손실**: 0% (Firestore 자동 백업)

**설계 원칙** (OurHome 실패 교훈):
- 완벽한 동기화 우선, 기능 추가는 이후
- 집안일 완료 → XP 지급 트랜잭션 원자성 보장
- 네트워크 오류 시 명확한 사용자 피드백

### 5.3 보안
- **인증**: Firebase Authentication
- **권한 관리**: Firestore Security Rules
- **데이터 암호화**: HTTPS, TLS 1.3
- **개인정보 보호**: GDPR, 한국 개인정보보호법 준수

### 5.4 사용성
- **학습 시간**: 5분 이내에 첫 집안일 완료 가능
- **접근성**: WCAG 2.1 AA 수준
- **다국어**: 한국어(기본), 영어(Phase 3)

### 5.5 확장성
- **수평 확장**: Firebase Auto-scaling
- **데이터베이스**: Firestore (수백만 문서 지원)
- **CDN**: Firebase Hosting + CDN

## 6. 성공 지표 및 측정

### 6.1 획득 지표 (Acquisition)
- 앱 스토어 다운로드 수
- 웹사이트 방문자 수
- 초대 코드 사용률

### 6.2 활성화 지표 (Activation)
- 온보딩 완료율 (가구 생성 + 첫 집안일 추가): > 70%
- 첫 집안일 완료 시간: < 10분

### 6.3 유지 지표 (Retention)
- Day 1 Retention: > 60%
- Day 7 Retention: > 40%
- Day 30 Retention: > 25%

### 6.4 참여 지표 (Engagement)
- 일일 활성 사용자(DAU) / 월 활성 사용자(MAU): > 0.3
- 가구당 일일 집안일 완료 수: > 3개
- 세션당 평균 시간: 3-5분

### 6.5 수익 지표 (Revenue)
- 유료 전환율: 5%
- 월 평균 매출(ARPU): ₩245 (5% × ₩4,900)
- 고객 생애 가치(LTV): ₩60,000 (연간 구독 가정)

### 6.6 추천 지표 (Referral)
- 초대당 신규 가입 수: > 1.5명
- 바이럴 계수(K-factor): > 1.0

### 7.1 기술적 제약
- Flutter Web 성능 한계 (모바일 네이티브 앱 대비)

**완화 방안**: 약관 및 개인정보처리방침 작성, 법률 자문

## 8. 의사결정 기록

### 8.1 왜 Flutter인가?
- 단일 코드베이스 → iOS/Android/Web 동시 지원
- 빠른 개발 속도 (Hot Reload)
- Firebase와의 뛰어난 통합

### 8.2 왜 Firebase인가?
- 실시간 동기화 (핵심 요구사항)
- 서버리스 아키텍처 → 초기 비용 절감
- 빠른 확장 가능 (Auto-scaling)

### 8.3 왜 프리미엄 모델인가?
- 기본 기능 무료 → 넓은 유입 경로
- 대가족/파워 유저만 유료 → 명확한 전환 동기
- Sweepy, Todoist 등 시장 검증된 모델

### 8.4 왜 카카오톡 연동인가?
- 한국 사용자의 95%가 카카오톡 사용
- 앱을 매일 열지 않아도 알림 수신 → 참여 장벽 하락
- 글로벌 경쟁사와의 결정적 차별화
